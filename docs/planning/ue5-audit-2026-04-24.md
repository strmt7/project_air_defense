## Landed Changes

### Main menu polish and attribution

`ProjectAirDefenseMainMenuWidget.cpp` received value-only refinements: title font 40 to 48, primary and secondary button label 24 to 26, warmer primary fill `(0.10, 0.26, 0.34)`, deeper secondary fill `(0.07, 0.08, 0.14)`, and a distinct chip palette where `LIVE 3D CITY` now reads green, `LANDSCAPE` reads blue, and `TOUCH HUD` reads warm. No new widget types, delegates, or signatures were introduced; the widget tree and binding contract are unchanged.

A required CC-BY 4.0 attribution line was added to the `CITY` footer panel: the label now reads `Helsinki Kalasatama 3D Tiles` followed by a smaller credit `City of Helsinki / Helsinki 3D  CC BY 4.0`. This satisfies the AGENTS.md and `android/assets/ATTRIBUTION.md` requirement to credit the City of Helsinki whenever the Helsinki 3D Tiles source is active.

### Simulation engagement-range geometry fix

`ProjectAirDefenseBattleSimulation::SelectNextThreat` used `Threat.PositionMeters.SquaredLength()` to gate engagement, which is a 3D distance from the world origin `(0,0,0)`. `bIsTracked` in `UpdateThreats` used a horizontal distance from `DefenseOriginMeters` (the mean of launcher positions). These two checks disagreed: a tracked threat at high altitude could be rejected for launch even though it was well inside the horizontal engagement cylinder centered on the launcher cluster.

`SelectNextThreat` now uses the same horizontal distance from `DefenseOriginMeters` that `bIsTracked` uses. Expected effect: a modest increase in engagement opportunities on cruise and glide threats that spend time at altitude, especially under the `ShieldWall` doctrine, which allows two interceptors per threat.

Test exposure: the existing `BattleSimulationLeakageTest` upper-bound of intercept rate below 0.98 is loose enough that this correction is not expected to break it, and the `DoctrineComparison` ordering is unaffected because the fix applies uniformly across doctrines. No existing test becomes incorrect; the fix only stops dropping valid targets.

### Procedural Patriot battery visuals

`ProjectAirDefenseBattleManager::SyncStaticVisuals` previously left `LauncherVisuals` empty after clearing it. Each launching station now renders as a composite M901 silhouette using only engine-primitive `CubeMesh` components:

- trailer deck: 10.5 m by 3.0 m by 1.2 m, aligned along the Y axis
- cab block: 3.2 m by 2.6 m by 2.8 m, placed forward of the trailer center toward the friendly city
- four PAC-3 canisters in a 2x2 cluster at the rear: each 0.72 m by 0.72 m by 6.0 m, elevated 20 degrees above horizontal, pointing toward the threat approach axis

Missile proportions in `ThreatMarkerRadiusMeters`, `ThreatMarkerLengthMeters`, `InterceptorMarkerRadiusMeters`, and `InterceptorMarkerLengthMeters` were rescaled from stubby bullet shapes to slender missile shapes (threat about 14 m by 1.8 m diameter, interceptor about 9 m by 1.1 m diameter). Game-scale length and diameter are roughly 1.5 to 2x real PAC-3 and SCUD-B class geometry to preserve readability at battery-overview camera distances while keeping a recognizable slender silhouette.

This is NOT synthetic city geometry; the launching stations are military hardware anchored to real launcher positions and complement the real Helsinki mesh rather than overlaying it. The existing ban on healthy synthetic city towers stays in force; the district beacon path remains suppressed.

The unused `LauncherMarkerRadiusMeters` and `LauncherMarkerHalfHeightMeters` constants were removed as dead code.

### Sub-step visual interpolation for high-refresh-rate displays

`ProjectAirDefenseBattleManager::SyncDynamicVisuals` now extrapolates each threat and interceptor marker forward by `StepAccumulatorSeconds`, the residual time between the last completed simulation step (typically 50 ms) and the current frame. Before this change, markers snapped to their stepped positions and held for every rendered frame inside the same simulation step, which produces visible micro-stutter at 120 Hz and above. Linear extrapolation using the most recent velocity vector is the standard remedy for fixed-step simulations driving variable-rate renderers; it has zero effect on the simulation itself and only improves how motion reads on fast displays.

### AGENTS.md policy: current-generation programming methods

A new `AGENTS.md` bullet makes it explicit that new UE5 C++ work must use C++20 features supported by UE 5.7 (structured bindings, `TOptional`, `if constexpr`, range-for, `auto` return deduction, `[[nodiscard]]`) and current-generation engine subsystems (CommonUI, EnhancedInput, World Partition, Nanite, Lumen, TSR, sub-step visual interpolation for high-refresh displays). Legacy Kotlin migration work must similarly prefer coroutines, sealed classes, inline/value classes, and flow-based state over callback ladders. Legacy patterns are not to be extended when a supported modern API exists.

## Simulation Review Notes

- `UpdateThreats` removes the current threat by `TArray::RemoveAt` without incrementing the index. That is correct because `RemoveAt` slides later elements down, and `Threat.Id` is copied into the event list before removal.
- Ballistic initial-velocity setup correctly compensates for gravity during the predicted travel time even though the `-=` line reads oddly (gravity is negative in this coordinate system).
- Guided-threat motion in `UpdateThreatMotion` renormalizes velocity back to `CruiseSpeedMetersPerSecond` after lerping toward the desired heading. Gravity for glide threats therefore behaves as a heading bias rather than a true speed loss. This is a deliberate arcade-leaning simplification and remains consistent with the doctrine comparison tests.
- ShieldWall is the only doctrine that allows two interceptors per threat. If two interceptors close within fuse in the same step, only the first commits the intercept; the second sees `FindThreat` return null and is culled. This is indirectly covered by `DoctrineComparisonTest`.
- Untested paths: cruise-only waves, launcher exhaustion under saturation, `InterceptorMaxAltitudeMeters` and `InterceptorMaxRangeMeters` culling, and mid-spawn wave handoff.

## Graphics Defaults

`UProjectAirDefenseGameUserSettings` defaults to TSR anti-aliasing, ambient occlusion on, motion blur off, ray tracing off, and quality level 3 across scalability groups with VSync off. That matches the mobile-first posture documented in `ue5-mobile-rendering`. No changes were made; downgrading defaults would reduce visual quality without measured need.

## Cross-Repo Review: ZMB-UZH/omero-docker-extended

The two repos already share most of the important structure: single-session rule, fast load order, `caveman` skill pinned and scoped, `context-budget`, `verification-loop`, pinned dependency versions, and an AGENTS.md at the root with a Claude adapter.

Findings on transferability:

- The omero AGENTS.md contains a strict AI commit identity rule (`AI agent <>` for author and committer, empty-email `Co-authored-by: AI agent`). The game repo currently ships human-author commits through `strmt7`. Adopting this policy here would change commit metadata for every AI-authored change; it is a policy decision the repo owner should make, not a silent port.
- The omero "Mandatory security read order" is domain-specific (filesystem paths, SQL, subprocesses, Docker). The game repo lacks comparable attack surface in its runtime module, so a verbatim port would add friction without matching risk.
- Several omero skills (`ai-regression-testing`, `tdd-workflow`, `search-first`, `docs-knowledge-maintainer`) name patterns already folded into the game repo's AGENTS.md ("Search first, patch second, benchmark third") or its existing `verification-loop` skill.
- The omero `frontend-preview` skill is Node and Django oriented and is not transferable to a UE5 packaged-runtime preview.

Net recommendation: do not port new skills or policies verbatim. If the repo owner later wants a formal AI commit identity, it should be added through an explicit AGENTS.md update rather than an implicit import from another project.
